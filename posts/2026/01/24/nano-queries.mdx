In June of 2023, I found myself in a picante situation. We used a custom database on a project, and I needed in a simple query builder to build complex queries dynamically and ensure security against user input. I found that there were no solutions to that problem on the JavaScript platform at all. I've built a solution and now open source it.

![](./uncut-gems-2019.jpg)

I've run a research and found that all packages to build queries on JavaScript platform had at least one of the fundamental problems below (usually most of them):

- They are built for a specific database or a few. They require "drivers", "compilers", etc that do not exist for our database
- They are designed to be ORM or anything but not just a query builder
- They work only on the Node platform, but not in the browser, Deno, etc

We had a custom database that run in browser, and we no need in ORM because it's slow by design [[1](https://medium.com/%40jholt1055/stop-using-orms-for-reporting-queries-theyre-100x-slower-than-raw-sql-db330f205828), [2](https://www.geldata.com/blog/why-orms-are-slow-and-getting-slower)] and makes difficult any performance tuning.

Meanwhile the most popular tools to work with a databases on JavaScript are ORMs: [Prisma](https://www.prisma.io/), [Kysely](https://kysely.dev/), [Drizzle](https://orm.drizzle.team/), [TypeORM](https://typeorm.io/), [Objection](https://www.npmjs.com/package/objection).

It seems a JavaScript programmers don't care about performance, and like to over engineer a simple things right?

## Definition of a Query Builder

The purpose of a **query builder** is to simplify the building of complex queries and ensure its safety. That's all.

You write a code in SQL or any other language that are native for your database. You have no surprises, and you can run `EXPLAIN ANALYZE` or equivalent against your queries to tune its performance specifically for needs of your project configuration.

## Nano Queries is a solution

That's why I have create a [`nano-queries`](https://github.com/vitonsky/nano-queries).

Nano queries is a query builder that consists of 3 primitives - a `Raw` segment that represent a raw code, `Value` that represent any user input, and a `Query` that represent a sequence of any of this three.

This simple design let represent safely a query of any complexity.

To make it applicable in real world projects, a Nano Queries provides helpers to compose these primitives via templated strings like that

```js
sql`SELECT title FROM movies WHERE release_year = ${2007}`;
```

This code build a `Query` segment that have 2 other segments, so an abstract data structure that represent a query above is looks like

```
Query[Raw["SELECT title FROM movies WHERE release_year = "], Value[2007]]
```

The queries are lazy, the SQL string and bindings will be build once user will compile a query and compiler will call a `getSegments` method of `Query`

```js
compile(sql`SELECT title FROM movies WHERE release_year = ${2007}`);

// Returns query with placeholders and array with bindings equal to
// {
//   sql: "SELECT title FROM movies WHERE release_year = $1",
//   bindings: [2007],
// }
```

This design does mean you can embed one `Query` into another, and modify queries conditionally after composing to implement a conditional query building.

```js
const userInput = {
  year: 2007,
  rating: 4.1,
};

// You may nest one query into another
// A `where` returns an instance of class that extends `Query` and provide `and` & `or` methods,
const filter = where();
const query = sql`SELECT title FROM movies ${filter} LIMIT 100`;

// A query segment can be extended any time before compiling
filter.and(sql`release_year = ${userInput.year}`);

// That's useful to build a complex conditional queries
if (userInput.rating > 0) {
  filter.and(sql`rating >= ${userInput.rating}`);
}

compile(query);
// {
//   sql: 'SELECT title FROM movies WHERE release_year = $1 AND rating >= $2 LIMIT 100',
//   bindings: [2007, 4.1],
// }
```

The query that we compile here may be represented as a data structure

```
Query[
  Raw["SELECT title FROM movies "],
  WhereClause[
    Query[Raw["release_year = "], Value[2007]],
    Query[Raw["rating >= "], Value[4.1]],
  ],
  Raw[" LIMIT 100"]
]
```

Notice the `WhereClause` that extends `Query` have no keywords `WHERE` & `AND`. Its method `getSegments` will insert all necessary segments depends on state and semantic.

In case no single query will be added to a `filter`, its method `getSegments` will return empty list and query will remain valid:

```sql
SELECT title FROM movies LIMIT 100
```

This design allow to implement a query segments with any semantic via add a new class based on `Query`. These segments may implement any SQL dialect, or even NoSQL.

If you frequently use some SQL features - just introduce a new query segment that implements it.

Let's say you want to insert many values into DB. It would be nice to have something like `set` that would receive a list of `Query` and would compile it to a SQL set:

```js
sql`INSERT INTO movies (title,release_year,rating) VALUES ${set(
  movies.map(
    (movie) => sql`(${movie.title}, ${movie.release}, ${movie.rating})`
  )
)}`
```

the result will be

```sql
INSERT INTO movies (title,release_year,rating) VALUES ($1, $2, $3), ($4, $5, $6)
```

## Try it on your project

We have used this solution in production since 2023 and [open-sourced](https://github.com/vitonsky/nano-queries) it for the community.

Nano queries work well with any database - SQLite, Postgres, MySQL, Oracle, GraphQL, PGLite, DuckDB, etc. Actually, you may build any queries that can be compiled to a string.
